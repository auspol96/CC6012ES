A **relationship** shows *how two or more entities are connected.*

## 4. Degree of a Relationship
The **degree** indicates how many entities are involved in a relationship:

| Degree | Description | Example |
|---------|--------------|----------|
| **Unary (1)** | One entity relates to itself | Employee supervises Employee |
| **Binary (2)** | Two entities are related | Student takes Course |
| **Ternary (3)** | Three entities are related | Doctor treats Patient in Ward |

ðŸ’¡ *Most relationships in databases are **binary** (two entities involved).*  

---

## 5. Cardinality (Multiplicity)
**Cardinality** describes *how many instances* of one entity can be linked to another.

| Relationship | Type | Meaning |
|---------------|------|----------|
| 1:1 | One-to-One | Each student has one ID card |
| 1:M | One-to-Many | One department has many lecturers |
| M:N | Many-to-Many | Students take many courses, and each course has many students |

ðŸ’¡ *Cardinality defines how data in one table connects to another.*

---

## 6. Steps in E-R Modeling
E-R modeling is part of the **database design process**, moving from concept to implementation.

1. **Develop the Logical Model**  
   Identify entities, attributes, and relationships.  
2. **Develop the Relational Model**  
   Convert the E-R diagram into relational tables with primary and foreign keys.  
3. **Generate the DDL (Data Definition Language)**  
   Write SQL `CREATE TABLE` statements to implement the design.

## 7. Example E-R Diagram (Text Representation)

```
+-------------+          TAKES           +-------------+
|   STUDENT   |------------------------->|   COURSE    |
|-------------|                         |-------------|
| Student_ID  |                         | Course_ID   |
| Name        |                         | Title       |
| Email       |                         | CreditHours |
+-------------+                         +-------------+
      1                                  M
```

**Explanation:**
- Each **Student** can take *many* courses (1:M relationship).  
- Each **Course** can be taken by *many* students (M:N relationship, implemented later with a linking table).  

This example demonstrates how entities (Student, Course) and relationships (TAKES) are represented before converting them into relational tables.
